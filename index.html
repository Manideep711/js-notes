EP 01
Execution Context in JavaScript
The fundamental container where all JS happens
🔑 Core Fundamental — Tattoo This On Your Brain
"Everything in JavaScript happens inside the Execution Context." — Akshay repeats this twice on purpose. It is the single most important sentence in all of JS.

What is an Execution Context?

Think of the Execution Context as a big box or container inside which all JavaScript code is executed. It has exactly two components:

Memory Component
Also called → Variable Environment
All variables and functions are stored here as key-value pairs before a single line of code runs.

Example: { a: undefined, fn: <code> }
Code Component
Also called → Thread of Execution
This is the thread in which the whole code is executed one line at a time. Nothing happens outside of this.
┌──────────────────────────────────────────────────────┐ │ EXECUTION CONTEXT │ │ │ │ ┌───────────────────────┐ ┌──────────────────────┐ │ │ │ MEMORY COMPONENT │ │ CODE COMPONENT │ │ │ │ (Variable Env.) │ │ (Thread of Execution)│ │ │ │ │ │ │ │ │ │ key → value │ │ line 1 executes ──► │ │ │ │ a → undefined │ │ line 2 executes ──► │ │ │ │ greet → { fn code} │ │ line 3 executes ──► │ │ │ │ │ │ ... │ │ │ └───────────────────────┘ └──────────────────────┘ │ └──────────────────────────────────────────────────────┘

Is JavaScript Synchronous or Asynchronous?

Core Fundamental
JavaScript is a synchronous, single-threaded language.
  • Single-threaded — JS can only execute one command at a time.
  • Synchronous — It executes in a specific order. It cannot move to the next line until the current line has finished. No jumping ahead.
Wait — what about AJAX (Asynchronous)?
You've heard of AJAX where 'A' stands for Asynchronous. Don't panic. JavaScript achieves async behaviour through browser APIs, the Event Loop, and the Callback Queue. The JS engine itself is always synchronous. We'll cover all that in depth in Episode 15.

EP 02
How JavaScript Code Is Executed
Two-phase process + the beautiful Call Stack

When you run any JS program the very first thing that happens is: a Global Execution Context (GEC) is created. It always comes first. No exceptions.

The Example Code

var n = 2;

function square(num) {
  var ans = num * num;
  return ans;
}

var square2 = square(n);   // invoke with n=2
var square4 = square(4);  // invoke with literal 4

The Two Phases of Execution Context Creation

Phase 1 — Memory Creation

JS skims the entire program before executing even line 1. It allocates memory to all variables and functions.

  • Variables → stored as undefined (a placeholder)
  • Functions → the entire code is literally copied into memory
Phase 2 — Code Execution

JS runs through the code again, line by line. This is where real values replace undefined, calculations happen, and functions are invoked.

Phase 1 Walkthrough — Memory Allocation

GLOBAL EXECUTION CONTEXT — Phase 1 (Memory) ┌──────────────────────────────────────────────┐ │ MEMORY │ │ │ │ n ──► undefined │ │ square ──► { var ans = num*num; return ans; }│ │ square2 ──► undefined │ │ square4 ──► undefined │ └──────────────────────────────────────────────┘

Phase 2 Walkthrough — Code Execution

Line 1: n = 2 → the undefined placeholder in memory is replaced with 2.

Lines 2–5: Function declaration — nothing to execute here, already stored in memory during Phase 1.

Line 6: square(n) is invoked. This is where it gets interesting.

🔑 Key Insight
Every single time a function is invoked, a brand-new Execution Context is created for it inside the Global Execution Context. Functions in JS are like mini-programs. Each invocation creates its own isolated world with its own memory and code components.

Inside square(n) — a new Execution Context is born:

square(n) Execution Context ┌────────────────────────────────────────┐ │ Phase 1 (Memory): │ │ num ──► undefined │ │ ans ──► undefined │ │ │ │ Phase 2 (Code): │ │ num ◄── 2 (argument passed) │ │ ans ◄── 2 × 2 = 4 │ │ return ans ──► 4 goes to square2 │ └────────────────────────────────────────┘ ↓ Execution Context is DELETED after return square2 = 4 ✓ Control returns to line 6

Then square(4) creates another new EC, computes 16, assigns to square4, and that EC is deleted. When the whole program finishes, the GEC is also deleted.

The Call Stack — JS's Traffic Controller

It is very difficult for the JS engine to manage all this execution context creation and deletion. But it does it very beautifully — using the Call Stack.

The Call Stack is a stack data structure (Last In, First Out — LIFO) that tracks which Execution Context is currently running.

During square(n) execution:
E1 — square(n) execution context ← currently running (TOP)
▲ pushed when square(n) was called
GEC — Global Execution Context ← always at the bottom

When square(n) returns → E1 is POPPED. Control back to GEC at line 6.
Other Names for the Call Stack
That's why JS is the most confusing language — everything has fancy names. Call Stack = Execution Context Stack = Program Stack = Control Stack = Runtime Stack = Machine Stack. All the same beautiful thing.

The Call Stack maintains the order of execution of Execution Contexts. When asked "what is the call stack?" — explain this mechanism: GEC pushed first, function ECs pushed on top when invoked, popped when returned, GEC popped when program ends.

Global Execution Context Memory Creation Phase Code Execution Phase Call Stack LIFO
EP 03
Hoisting in JavaScript
Accessing variables and functions before they're declared — the magic explained
Definition
Hoisting is a phenomenon in JavaScript by which you can access variables and functions even before you have initialized them — without getting an error.

The Magic Demo

// These lines are BEFORE the declarations — and they work!
getName();          // → "Namaste JavaScript" ✓
console.log(x);    // → undefined  (not an error!)
console.log(getName); // → prints the entire function code

var x = 7;
function getName() {
  console.log("Namaste JavaScript");
}

In other languages this would be an error. In JS it works because of Phase 1 — memory was already allocated for x (undefined) and the full code of getName was already copied into memory before line 1 even executed.

undefined vs "not defined" — They Are NOT the Same

TermWhat it meansWhen you see it
undefinedMemory was allocated, but no value assigned yet. It IS in memory.var declared but not yet assigned
not defined (ReferenceError)No memory was allocated. The identifier doesn't exist anywhere.Variable never declared in the program
// Remove "var x = 7" entirely from the program:
console.log(x);  // ReferenceError: x is not defined
// Because now x was NEVER in memory at all

How Different Function Syntaxes Hoist

SyntaxHoisted asCan call before declaration?
Function Declaration
function foo() {}		Full function code ✓Yes ✓
Function Expression
var foo = function() {}		undefinedNo — TypeError ✗
Arrow Function
var foo = () => {}		undefinedNo — TypeError ✗
Arrow functions and function expressions behave just like another variable. So if you call them before their line, JS sees undefined and says "you're trying to call undefined as a function?" — TypeError.

Don't say: "Hoisting moves code to the top of the file."
Say this: "Hoisting happens because of the Memory Creation Phase (Phase 1) of the Execution Context. Before any code executes, JS allocates memory to all variables (storing undefined) and copies the entire code of function declarations. That's why we can access them before their line in the source code."

EP 04
Functions & Variable Environments
Each invocation = its own isolated world. Demonstrated in DevTools.

This episode shows — live in the browser debugger — that every function invocation creates its own execution context with its own completely isolated variable environment.

Variable Environment Isolation

var x = 1;

function a() {
  var x = 10;  // local x — completely separate from global x
  console.log(x); // 10
}

function b() {
  var x = 100;
  console.log(x); // 100
}

a();
b();
console.log(x); // 1 — global x untouched
Call Stack during a() execution: ┌──────────────────────────────────┐ │ a()'s EC → x: 10 │ ← TOP └──────────────────────────────────┘ ┌──────────────────────────────────┐ │ GEC → x: 1, a: fn, b: fn │ ← BOTTOM └──────────────────────────────────┘ x inside a() is COMPLETELY different from x in GEC

Seeing It Live — DevTools Demo

Put a debugger; statement inside a function. In Chrome DevTools → Sources panel → when execution pauses, look at the Call Stack panel on the right. You'll see:

  • anonymous = the Global Execution Context (always at the bottom)
  • Function name = pushed on top when the function runs
  • When the function returns → it disappears (popped off)
  • Once everything is done → Call Stack is empty
Pro tip
Open DevTools → Sources → add debugger; in your code → refresh. Step through with F10. Watch variables change in the Scope panel. This is the execution context made visible.
EP 05
The Shortest JS Program — window & this
What happens when you run a completely empty JS file?

The shortest JavaScript program is an empty file. Even with zero lines of code, when the JS engine runs it: a Global Execution Context is created AND a global object is created.

Key Concept
Along with the GEC, the JS engine always creates a global object. In browsers it's called window. In Node.js it's called global. The name changes by environment — but there is always a global object. This is a responsibility of the JS engine itself.

window, this, and the Global Object

// In browser — even in an empty file:
console.log(window);  // → Window { ... } — the global object
console.log(this);    // → Window { ... } — same thing at global level

this === window;      // → true

Global Variables Attach to window

Any var variable or function declaration in the global space (not inside any function) gets attached to the window object automatically.

var a = 10;
function greet() { console.log("hi"); }

window.a;      // 10 — attached!
window.greet;  // ƒ greet() — attached!

// You can even access without window. prefix:
a;        // 10
greet();  // "hi"
Why this matters later
Understanding that global vars become properties of window is critical for understanding global namespace pollution, and why let/const (which do NOT attach to window) are preferable in modern code.

EP 06
undefined vs not defined
A very special keyword unique to JavaScript — and a mistake you must never make
Core Concept
undefined is NOT the absence of a value. It is a real keyword that takes up memory space. It's the placeholder JS deliberately puts into a variable's memory slot during Phase 1, before any value is assigned.

The Lifecycle of a Variable

// Phase 1 allocates memory for 'a' → placeholder: undefined
// Phase 2 executes line by line

console.log(a);  // undefined — memory exists, value not yet
var a = 7;
console.log(a);  // 7 — placeholder replaced with actual value

undefined ≠ Empty

undefined is not empty. It is a concrete value — a special keyword — that JS puts in a slot to say: "memory is reserved, but nothing has been assigned yet." You can even check for it:

var a;
if (a === undefined) {
  console.log("a is undefined"); // runs
}
a = 10;
if (a === undefined) {
  console.log("a is undefined");
} else {
  console.log("a is not undefined"); // runs now
}

JavaScript is a Loosely Typed Language

Also called a weakly typed language. JS doesn't bind variables to a specific data type — you can reassign freely from string to number to boolean. This is not a weakness:

JavaScript is not weak at all. In fact I feel the language is more stronger — because it is handling a lot of stuff behind the scenes. You give it a string, it handles automatically. You give it a number, it takes it. That's flexibility, not weakness.
var a;              // undefined
a = 10;            // number — fine
a = "hello";      // string — also fine!
a = true;          // boolean — perfectly valid JS
⚠️ Never Do This Mistake
Never manually assign undefined to a variable: var a = undefined;

undefined is a sentinel value JS uses to communicate "not yet assigned." Polluting that signal leads to confusing bugs. If you want to empty a variable, use null. Let undefined serve its original purpose.

EP 07
Scope Chain & Lexical Environment
How JavaScript finds variables — the chain of nested environments
Core Fundamental
Scope in JavaScript is directly related to the Lexical Environment.

What is Scope?

Scope answers two related questions:

  • "Where can I access this variable?" — this is the scope of the variable
  • "Is this variable accessible from here?" — is it inside the scope?

The Code Example

function a() {
  function c() {
    console.log(b); // 10 — can access b from outer scope!
  }
  var b = 10;
  c();
}
a();

console.log(b); // ReferenceError: b is not defined
// b is not accessible in global scope

What is Lexical Environment?

Definition — Remember This Forever
Lexical Environment = Local Memory + Reference to the Lexical Environment of its Parent.

"Lexical" means in order, in hierarchy, by physical location in the source code. Where a function sits in the code determines its lexical parent.

In the example: c is physically inside a → c's lexical parent is a. a is physically inside global → a's lexical parent is global. Global's parent → null.

GLOBAL EXECUTION CONTEXT ┌──────────────────────────────────────────────────────────┐ │ Memory: { a: fn } │ │ Lexical Parent Reference: null │ │ │ │ A's EXECUTION CONTEXT │ │ ┌────────────────────────────────────────────────────┐ │ │ │ Memory: { b: 10, c: fn } │ │ │ │ Lexical Parent Reference: ──────────────────► GEC │ │ │ │ │ │ │ │ C's EXECUTION CONTEXT │ │ │ │ ┌──────────────────────────────────────────────┐ │ │ │ │ │ Memory: {} │ │ │ │ │ │ Lexical Parent Reference: ────────────► A │ │ │ │ │ └──────────────────────────────────────────────┘ │ │ │ └────────────────────────────────────────────────────┘ │ └──────────────────────────────────────────────────────────┘

How JS Searches for a Variable

When c does console.log(b):

  1. JS checks c's local memory → b not found
  2. Follows lexical parent reference → goes to a's memory → found! Returns 10

If b wasn't in a's memory either:

  1. Goes to a's parent → Global memory → searches there
  2. Still not found → Global's parent → null → search ends
  3. Throws: ReferenceError: b is not defined

The Scope Chain

Definition
The Scope Chain is this entire chain of lexical environments — local → parent → grandparent → … → null. It determines what is and isn't accessible from any given point in your code. The chain of all these lexical environment references is the scope chain.
Sneak Peek at Closures
In DevTools, when paused at a breakpoint inside c, you'll see a Closure entry in the Scope panel containing b from a. This is closure! c has enclosed a's variables in its scope. We cover this fully in Episode 10.

EP 08
let, const & Temporal Dead Zone
ES6 variables and why they behave very differently from var
Key Insight — Yes, let and const ARE Hoisted
let and const are hoisted — memory IS allocated for them in Phase 1. BUT they are stored in a separate memory space (not on the global object), and they cannot be accessed before the line where they are declared. This period of inaccessibility is the Temporal Dead Zone (TDZ).

The Temporal Dead Zone Explained

Timeline for: let b = 20; Program start │ ▼ [Phase 1: Memory allocated for b — in SEPARATE SPACE, not window] │ │ ◄────────────── TEMPORAL DEAD ZONE ──────────────► │ b exists in memory BUT is inaccessible │ Accessing b here → ReferenceError │ ▼ let b = 20; ← TDZ ENDS here. b is now accessible with value 20 │ ▼ console.log(b); // 20 ✓
console.log(a); // undefined — var is accessible (just has placeholder)
console.log(b); // ReferenceError: Cannot access 'b' before initialization

var a = 10;
let b = 20;  // b exits TDZ here

Three Types of Errors with let & const

SyntaxError
Caught before code runs
Re-declaring a let variable, or declaring const without initialization.
TypeError
At runtime
Trying to re-assign a const variable after it's been set.
ReferenceError
At runtime
Accessing a let/const variable while it's in the Temporal Dead Zone.
// SyntaxError — caught before any line runs:
let a = 10;
let a = 20; // SyntaxError: Identifier 'a' already declared
const b;   // SyntaxError: Missing initializer in const declaration

// TypeError — at runtime:
const c = 100;
c = 200;   // TypeError: Assignment to constant variable

// ReferenceError — TDZ:
console.log(d); // ReferenceError: Cannot access 'd' before init
let d = 10;

How to Avoid the Temporal Dead Zone

Best Practice
Always declare and initialize your variables at the very top of their scope. This shrinks the TDZ to zero — the moment the declaration line is hit, it's immediately initialized. No gap, no danger.

Which Should You Use?

  • Use const whenever possible — for values that don't change.
  • Use let when you need to reassign.
  • Avoid var in modern code — no block scope, pollutes window, confusing hoisting.

"Are let and const hoisted?" → Yes, they are hoisted, but they're kept in the Temporal Dead Zone from the start of their scope until the line where they are initialized. Accessing them before that line throws a ReferenceError. Unlike var, they are not stored in the global object — they live in a separate memory space."

EP 09
Block Scope & Shadowing in JavaScript
What a block is, how let/const live inside it, and illegal shadowing rules

What is a Block?

Definition
A block (also called a compound statement) is code wrapped in curly braces { }. It's used to group multiple JS statements together in places where JS expects only one statement (like after if, for, etc.). 
// Block scope in action:
{
  var a = 10;   // goes to global/function scope — NOT block-scoped
  let b = 20;   // block-scoped — only lives inside { }
  const c = 30; // block-scoped — only lives inside { }
}

console.log(a);  // 10 ✓ — var leaks out of blocks
console.log(b);  // ReferenceError — let is gone
console.log(c);  // ReferenceError — const is gone

During Phase 1 for the block, b and c are stored in a separate block-scope memory — not in global. a goes into global memory as always.

Shadowing

When a variable inside a block has the same name as one outside, the inner one shadows the outer one within the block. The behaviour differs between var and let:

var shadowing — modifies original!
var a = 100;
{
  var a = 10; // SAME memory location!
  console.log(a); // 10
}
console.log(a); // 10 — global was CHANGED
let shadowing — safe, separate!
let b = 100;
{
  let b = 20; // DIFFERENT memory location
  console.log(b); // 20
}
console.log(b); // 100 — outer b untouched ✓

Illegal Shadowing

Rule
You cannot shadow a let variable with var inside a block. But you can shadow a var with let, or let with let. 
let a = 10;
{
  var a = 20;  // ✗ SyntaxError: 'a' already declared
  // Reason: var tries to go to global scope, but let a is already there
}

// This IS fine:
var b = 10;
{
  let b = 20;  // ✓ OK — let b lives in block, no conflict
}

"What is a block in JavaScript?" → A block is code enclosed in { } used to group multiple statements. let and const are block-scoped — they live only inside the block they're declared in. var ignores blocks and goes to the nearest function or global scope.

EP 10
Closures in JavaScript
The most beautiful and powerful concept in JS
🔑 Official Definition
A closure is a function bundled together with its lexical environment. In other words, a function along with its outer scope — even when that function is executed far outside where it was originally defined.
Functions are so beautiful that they remember their lexical environment wherever they go. Even when a function is returned from another function and executed in a completely different scope — it still has access to the variables of its original scope. That's the magic of closures.

Basic Example

function x() {
  var a = 7;

  function y() {
    console.log(a); // can access a — closure!
  }

  y();
}
x(); // → 7

The Real Power — Functions Returned from Functions

function x() {
  var a = 7;

  function y() {
    console.log(a);
  }

  return y;  // return the function itself (not its result)
}

var z = x();  // x() executes, returns y, x's EC is deleted
z();          // → 7  ← a still accessible even though x is gone!
What just happened?
When x() finished executing, its Execution Context was deleted. Yet when we call z() — which is the returned function y — it still prints 7. Why? Because y closed over the variable a from its lexical environment. The closure kept that reference alive.
Critical — Closures hold REFERENCES, not values
A closure remembers a reference to the variable, not a snapshot of its value at the time of creation. This distinction is crucial for understanding the famous setTimeout loop bug (covered in Episode 11).

Compact Syntax — How Developers Write Closures

function x() {
  var a = 7;
  return function y() {  // return directly, more concise
    console.log(a);
  }
}
x()();  // → 7  (double parentheses — call x, then call the returned fn)

Uses of Closures

  • Data Hiding & Encapsulation — private variables (see Episode 12)
  • Module Design Pattern — currying, partial application
  • setTimeout callbacks — they remember their outer scope
  • Event listeners — the handler closes over counter variables etc.
  • memoization — caching results in a closure
  • Iterators & generators — state is maintained via closure
  • once() function — execute something only one time

Classic interview question: "What is a closure?" → Say: "A closure is a function bundled together with its lexical environment. When a function is returned from another function, it carries a reference to its outer scope's variables — even after the outer function's execution context has been deleted. Those variables are kept alive by the closure."

EP 11
setTimeout + Closures
The interview question that makes senior developers pull their hair out
Time and tide wait for none — and just like that, JavaScript also waits for none. JavaScript does not wait for setTimeout. It registers the callback, attaches a timer, and moves on.

How setTimeout Actually Works

function x() {
  var i = 1;
  setTimeout(function() {
    console.log(i); // this callback forms a closure over i
  }, 1000);
  console.log("Namaste JavaScript");
}
x();
// Output:
// "Namaste JavaScript"  ← prints immediately
// 1                     ← prints after 1 second

What setTimeout does: takes the callback function → attaches a timer → stores it somewhere → JS moves on immediately without waiting → when the timer expires, the callback is put into the Callback Queue → Event Loop picks it up → runs it.

The Famous Interview Question — print 1 through 5 with delays

// ✗ WRONG — what most developers write first:
for (var i = 1; i <= 5; i++) {
  setTimeout(function() {
    console.log(i);
  }, i * 1000);
}
// Output: 6 6 6 6 6  ← WHY?!
Why does it print 6 five times?
Because all five callback functions close over the SAME i variable reference. When the loop runs, JS doesn't wait for any timer — it rushes through all 5 iterations, incrementing i to 6. By the time any timer fires and the callbacks execute, i is already 6 in that shared memory location.

Solution 1 — Use let (block-scoped)

// ✓ Fix 1 — let creates a NEW i for each iteration:
for (let i = 1; i <= 5; i++) {
  setTimeout(function() {
    console.log(i); // each callback closes over its own i
  }, i * 1000);
}
// Output: 1 2 3 4 5 (each after 1, 2, 3, 4, 5 seconds)

With let, each loop iteration creates a new copy of i in its own block scope. Each callback closes over its own separate i.

Solution 2 — Use Closures with var (the hard way, but shows mastery)

// ✓ Fix 2 — when interviewer says "use var only":
for (var i = 1; i <= 5; i++) {
  function close(x) {        // new function creates new scope
    setTimeout(function() {
      console.log(x);        // x is a fresh copy each time
    }, x * 1000);
  }
  close(i);  // pass i as argument — creates new copy
}
// Output: 1 2 3 4 5 ✓

This is the gold standard closure interview question. If an interviewer asks you to do it with only var and you can reach for a closure function to create a new scope — you've demonstrated deep understanding. Make closures work for you, not against you.

EP 12
Closure Interview Deep Dive
Data hiding, encapsulation, garbage collection, and the crazy interview questions

Advantages of Closures

  • Data hiding and encapsulation
  • Module design pattern
  • Currying and partial application
  • memoization
  • Maintaining state in async operations

Data Hiding with Closures

Suppose we want a counter. If we use a global variable, anybody can modify it from anywhere. But with closures we can hide that data — nobody from outside can access the counter directly.
// Problem with global variable — anyone can tamper:
var count = 0;
function increment() { count++; }
// Bad: someone can do count = 1000; from anywhere

// Solution — use closure for data privacy:
function counter() {
  var count = 0;  // private — no one outside can touch this

  return function increment() {
    count++;
    console.log(count);
  }
}

var counter1 = counter(); // independent counter
var counter2 = counter(); // another independent counter

counter1(); // 1
counter1(); // 2
counter2(); // 1 ← completely separate closure, its own count

Each call to counter() creates a fresh closure with its own private count. They don't interfere with each other.

Disadvantages of Closures

Memory Consumption
Closures keep variables alive even after the outer function returns. If you have too many closures, or if they're holding large objects, those variables won't be garbage collected — leading to memory leaks.

What is a Garbage Collector?

The Garbage Collector is a program in the browser (or JS runtime) that frees up memory that is no longer being used. In languages like C/C++, developers had to manually free memory. In JS, the engine does it automatically.

V8's Smart Garbage Collection
Modern JS engines like V8 are very smart about closures. If a variable inside a closure is never actually referenced by the inner function, V8 may garbage collect it even though the closure is active. For example, if a closure has access to both x and y but only ever uses x, V8 can collect y.

Closure Gotchas in Interviews

// Q: Do let declarations close over?
function outer() {
  let a = 10;
  return function inner() {
    console.log(a);
  }
}
outer()(); // → 10 ✓ Yes, let and const also close over

// Q: Do function parameters close over?
function outer(str) { // parameter = also part of lexical env
  return function inner() {
    console.log(str);
  }
}
outer("hello")(); // → "hello" ✓ Yes, parameters close over too

The event loop question: if you attach an event listener to a button and inside the handler you reference a counter variable via closure — that closure keeps the counter alive as long as the button exists on the page. To free memory, you should remove the event listener when it's no longer needed using removeEventListener.

EP 13
First-Class Functions in JavaScript
Function statement · expression · declaration · anonymous · named · arrow
I was giving an interview at a very reputed Indian startup — a famous online food ordering app. The interviewer asked me: "What are first-class functions?" I got nervous. I had never heard this term before. I said I don't know. And I knew the interview was gone.

Function Statement (= Function Declaration)

The most basic way to create a function. Also called a function declaration — these two terms mean exactly the same thing.

function a() {
  console.log("function a called");
}
a(); // works

Function Expression

A function assigned to a variable. Functions act like values in JS — you can assign them.

var b = function() {
  console.log("function b called");
};
b(); // works

Key Difference: Hoisting

The Major Difference
Function statements are hoisted (full code in memory). Function expressions are NOT hoisted — treated as variables, stored as undefined until their line executes. 
a(); // ✓ works — hoisted fully
b(); // ✗ TypeError: b is not a function — undefined until line 5

function a() { console.log("a"); }
var b = function() { console.log("b"); };

Anonymous Functions

Callback Functions & Event Listeners
How async operations are possible in a single-threaded language
Definition
A callback function is a function passed as an argument to another function, to be called later — either after some time, or when a certain event occurs. 
function x(y) {  // y is the callback
  console.log("x is executing");
  y(); // calling the callback
}

x(function y() {
  console.log("y was called back");
});

Callbacks Give JS Superpowers

JavaScript is a single-threaded language. It can only do one thing at a time. But callbacks give us the power of asynchronous programming — we can register something to happen later, and JS moves on without waiting.
setTimeout(function() {
  console.log("runs after 5 seconds");
}, 5000);
console.log("this runs immediately");
// Output:
// "this runs immediately"
// "runs after 5 seconds" (5 seconds later)

Event Listeners with Closures

function attachEventListener() {
  let count = 0; // private thanks to closure
  document.getElementById("btn")
    .addEventListener("click", function() {
      count++;               // remembers count via closure
      console.log("clicked", count);
    });
}
attachEventListener();

Blocking the Main Thread — Don't Do This

⚠️ Important Warning
JS has only one Call Stack. If you run a very heavy operation (like a 10-million loop) synchronously, the entire browser freezes — no clicks, no scrolls, nothing responds. This is called blocking the main thread. Always use async patterns (callbacks, promises, etc.) for heavy work.

Garbage Collection & Event Listeners

Event listeners are heavy — they hold closures in memory. If you attach thousands of event listeners and never remove them, memory accumulates. Always use removeEventListener when you no longer need the handler.

The Event Loop

Definition
The Event Loop has one job: continuously monitor the Call Stack and the Callback Queue. When the Call Stack is empty AND there's a callback waiting in the queue → the Event Loop picks the callback and pushes it into the Call Stack to execute.

The Callback Queue

When a timer expires, or an event fires, the callback goes into the Callback Queue (also called Task Queue / Macrotask Queue). The Event Loop picks from here when the Call Stack is empty.

The Microtask Queue — Higher Priority

Very Important — Asked in Interviews
There is a Microtask Queue that has higher priority than the Callback Queue. Callbacks from Promises and MutationObserver go into the Microtask Queue. The Event Loop always drains the entire Microtask Queue before picking anything from the Callback Queue. 
setTimeout(function() {
  console.log("setTimeout callback"); // Callback Queue
}, 0);

Promise.resolve().then(function() {
  console.log("Promise callback");    // Microtask Queue
});

console.log("synchronous");
// Output:
// "synchronous"
// "Promise callback"   ← Microtask Queue runs first!
// "setTimeout callback" ← Callback Queue runs after

Starvation of Callback Queue

If the Microtask Queue keeps generating new microtasks (e.g., a promise that resolves another promise that resolves another…), the Callback Queue tasks may never get to execute. This is called starvation.

"Explain the Event Loop." → "The Event Loop monitors the Call Stack and the Callback Queue. When the Call Stack is empty, it picks tasks from the Microtask Queue first (promises, MutationObserver), then from the Callback Queue (setTimeout, events). The Microtask Queue has higher priority."

JavaScript Engine, Runtime & V8
What actually happens inside the V8 engine when your code runs

JavaScript Runtime Environment

Definition
A JavaScript Runtime Environment is a big container with everything needed to run JS code: the JS Engine, APIs, Event Loop, Callback Queue, and Microtask Queue. Without a runtime environment, JS is impossible to run.
Browser Runtime
window object wraps all browser APIs. V8 (Chrome), SpiderMonkey (Firefox), JavaScriptCore (Safari)
Node.js Runtime
Open-source runtime environment. Uses V8 inside but replaces browser APIs with Node-specific APIs (fs, http, etc.)

The JS Engine — Common Myths

A lot of developers get confused and think the JS engine is some magical machine. No — it is just a normal program! A program written to execute JavaScript code. It is not a physical machine, not hardware — just software written in some language (like C++).

Inside the V8 JS Engine

When you give your code to the JS engine, three major steps happen:

1. Parsing
Code is broken into tokens, then parsed into an Abstract Syntax Tree (AST) — a structured tree representation of your code.
2. Compilation
The AST is compiled. Modern JS engines use Just-In-Time (JIT) compilation — both interpreter and compiler work together.
3. Execution
The code executes using the Call Stack (Memory Heap + Call Stack). The Garbage Collector works alongside.

Is JavaScript Interpreted or Compiled?

Interesting Interview Question
JavaScript was originally designed as an interpreted language. But today, modern browsers use Just-In-Time (JIT) compilation — a mix of both. The interpreter runs the code fast without waiting for full compilation, while the compiler simultaneously optimizes hot code paths. It's both, not just one.

JIT Compilation in V8

Your Code (JS) │ ▼ Parsing → AST (Abstract Syntax Tree) │ ▼ Ignition Interpreter → produces Bytecode (executes fast) │ ▼ (for frequently-run "hot" code) TurboFan Compiler → produces optimized Machine Code │ ▼ Execution at machine speed ⚡

Garbage Collector — Mark & Sweep

JS engines use the Mark and Sweep algorithm. The GC marks all variables that are reachable from the root (global object), then sweeps and frees anything that was NOT marked. This is how memory is automatically managed.

Trust Issues with setTimeout
Why setTimeout(fn, 5000) does NOT always run after exactly 5 seconds
⚠️ setTimeout Does NOT Guarantee Exact Timing
setTimeout(fn, 5000) does not guarantee the callback runs after exactly 5 seconds. It means the callback will run after at least 5 seconds — depending on what the Call Stack is doing.

Why the Delay Can Be Longer

setTimeout(function() {
  console.log("callback");
}, 5000);

// Blocking code that takes 10 seconds:
let startDate = new Date().getTime();
let endDate = startDate;
while (endDate < startDate + 10000) {
  endDate = new Date().getTime();
}
// "callback" runs after ~10 seconds, not 5!
// Because the while loop blocks the Call Stack for 10 seconds.
// The callback was waiting in the queue during that time.
The Event Loop will only give the callback a chance to execute when the Call Stack is empty. If the Call Stack is occupied for 10 seconds with blocking code, the callback waits even though its timer expired at 5 seconds. The timer is a minimum, not a guarantee.

setTimeout(fn, 0) — What Does It Mean?

setTimeout(function() {
  console.log("setTimeout 0");
}, 0);
console.log("sync");
// Output:
// "sync"           ← runs first (Call Stack)
// "setTimeout 0"   ← runs after (Callback Queue, after stack is empty)


setTimeout(fn, 0) doesn't mean "run immediately." It means "run as soon as the Call Stack is empty." It still goes through the Callback Queue mechanism. A delay of 0ms is the minimum possible delay, not instant execution.



Concurrency Model in JavaScript


JS achieves concurrent-like behaviour through its Event Loop + Web APIs + Callback Queue pattern. Even though JS is single-threaded, the browser can do multiple things (timers, network calls, event listeners) in parallel — the JS engine just picks up the results when it's free.

Higher-Order Functions & Functional Programming
How to write interview-quality code + the DRY principle
Definition
A Higher-Order Function is a function that either:
  • Takes another function as an argument (callback), OR
  • Returns a function from it

The Problem — Copy-Paste Programming

In interviews, I have seen a lot of people just copy-pasting. They write the formula again and again in three different functions. That is not good code. The interviewer is testing whether you can think modularly.
// ✗ BAD — DRY violation (Don't Repeat Yourself):
const radius = [1, 2, 3];

const calculateArea = (radius) => {
  const output = [];
  for (let i = 0; i < radius.length; i++) {
    output.push(Math.PI * radius[i] ** 2); // logic repeated below
  }
  return output;
};
const calculateCircumference = (radius) => {
  const output = [];
  for (let i = 0; i < radius.length; i++) {
    output.push(2 * Math.PI * radius[i]); // 90% same code!
  }
  return output;
};

The Solution — Functional Style with Higher-Order Functions

// ✓ GOOD — extract logic, pass it in:
const area = (r) => Math.PI * r ** 2;
const circumference = (r) => 2 * Math.PI * r;
const diameter = (r) => 2 * r;

// One generic higher-order function — the only loop:
const calculate = (radius, logic) => {
  const output = [];
  for (let i = 0; i < radius.length; i++) {
    output.push(logic(radius[i]));  // call the logic callback
  }
  return output;
};

console.log(calculate(radius, area));
console.log(calculate(radius, circumference));
console.log(calculate(radius, diameter));
This IS a Polyfill of map()
The calculate function above is almost exactly what Array.prototype.map() does internally. Each unit function (area, circumference, diameter) has exactly one responsibility. This is the essence of functional programming.

Principles Demonstrated

  • DRY — Don't Repeat Yourself. The loop is written once.
  • Single Responsibility — each function does exactly one thing.
  • Composability — mix and match logic functions freely.
  • Modularity — small, testable, reusable units.

In your coding round interview: don't just solve the problem — solve it modularly. Break down your logic into small functions, pass the logic as callbacks. This shows the interviewer you understand functional programming and clean code principles.

EP 19
map · filter · reduce
The three most important higher-order functions in JavaScript

Array.map() — Transform Each Element

When to use map
Use map when you want to transform each element of an array and get a new array of the same length with the transformed values. 
const arr = [5, 1, 3, 2, 6];

// Double each value:
const double = (x) => x * 2;
console.log(arr.map(double)); // [10, 2, 6, 4, 12]

// Triple — inline arrow function:
console.log(arr.map(x => x * 3)); // [15, 3, 9, 6, 18]

// Convert to binary:
console.log(arr.map(x => x.toString(2))); // ["101","1","11","10","110"]

Array.filter() — Keep Elements That Pass a Test

When to use filter
Use filter when you want to filter elements based on a condition. Returns a new array with only elements where the callback returns true. 
const arr = [5, 1, 3, 2, 6];

// Only odd numbers:
console.log(arr.filter(x => x % 2 !== 0)); // [5, 1, 3]

// Only even numbers:
console.log(arr.filter(x => x % 2 === 0)); // [2, 6]

// Only values greater than 4:
console.log(arr.filter(x => x > 4));          // [5, 6]

Array.reduce() — Reduce to a Single Value

When to use reduce
Use reduce when you need to iterate over all elements and produce a single value — a sum, a max, an object, anything. The accumulator holds the running result.
If I was given the option, I would never name this function "reduce" — because it's so confusing. But it's called reduce, so here we are. Just think of it as: iterate + accumulate → single output.
const arr = [5, 1, 3, 2, 6];

// Sum of all elements:
const sum = arr.reduce((acc, curr) => acc + curr, 0);
console.log(sum); // 17
//  acc starts at 0 (initial value)
//  each iteration: acc = acc + curr

// Maximum element:
const max = arr.reduce((acc, curr) => {
  if (curr > acc) acc = curr;
  return acc;
}, 0);
console.log(max); // 6

Advanced Example — Count Occurrences (reduce → object)

const users = [
  { firstName: "Akshay", lastName: "Saini", age: 26 },
  { firstName: "Donald", lastName: "Trump", age: 75 },
  { firstName: "Elon", lastName: "Musk", age: 50 },
  { firstName: "Deepika", lastName: "Padukone", age: 26 },
];

// Count users per age group:
const ageCount = users.reduce((acc, curr) => {
  if (acc[curr.age]) {
    acc[curr.age]++;
  } else {
    acc[curr.age] = 1;
  }
  return acc;
}, {}); // initial value is empty object
console.log(ageCount); // { 26: 2, 75: 1, 50: 1 }

Chaining map + filter + reduce

// First names of all users under 30:
const youngFirstNames = users
  .filter((x) => x.age < 30)      // keep age < 30
  .map((x) => x.firstName);        // extract firstName

console.log(youngFirstNames); // ["Akshay", "Deepika"]
